# グラウンディング検証プロンプト（Step 8.6）

Step 8 / 8.5 で生成した各 item の文章・図・X 投稿文が、**その記事の一次情報だけに基づいている**ことを保証する。ハルシネーション（記事に無い数値・主張の混入）を **書き込み前** に止めるための検証。

> なぜ必要か: 要約・図・`x_post` は「記事内の事実のみ」が原則だが、これまで自動検証が無かった。
> 特に Top Picks の `x_post` は **X に自動投稿される**ため、捏造数値は公開され取り返しがつかない。

## 重要: いつ実行するか

- **生成と同じターン内**（記事の `raw_excerpt` / `summary_en` がまだ文脈に残っている間）に行う。
- **書き込み前の自己チェック**として行う。すでに書いた JSON を再生成するのではなく、Edit で
  書き込む内容を確定させる前に、各 item の主張を点検して直す。**二重出力しない**（トークン節約）。

## 検証手順（item ごと）

1. **抽出**: その item の `summary_ja` / `key_points_ja` / `figure.data`（`headline` / `tldr` /
   `value` / `delta` / `before` / `after` / `pct` / `narrative` / `impact` / `points[].description` /
   `events[].description` 等のテキスト全部）/ `x_post` から、次を列挙する:
   - 数値・割合・倍率・価格・日付・期間
   - ベンチマーク名・モデル名・製品名・企業名・人名
   - 「世界初」「最速」「SOTA」「○○を上回る」等の評価・比較の断定
2. **照合**: 各項目が、その記事の `raw_excerpt` / `summary_en` に **明示されている、または明確に含意される**かを確認する。
3. **判定と修正**（記事に裏付けが無い / 食い違う場合）:
   - **書き直す** — grounded な表現に置き換える（例: 記事に無い「2 倍」を「向上」に）
   - **落とす** — その数値・固有名詞・断定を削除する
   - **図を降格** — 数値依存の `comparison` / `metric-bars` が成立しないなら `summary-card` に変える
     （`summarize-ja.md` のフォールバック方針に従う。記事内の質的事実で構造化）
   - **創作・推測・記事外知識での補完は禁止**（迷ったら落とす）

## ティア（厳しさの段階）

- **Top Picks ＋ `x_post` = 最厳・ゼロトレランス**: `x_post` の全語が、grounded な
  `summary_ja` / `key_points_ja` に辿れること。1 つでも裏付け不能な主張があり安全化できなければ、
  **その item の `x_post` を生成しない**（空のまま。失敗ではなく正常な結果）。
- **その他の item = 重点スキャン**: invented number（記事に無い数値）/ 誇張表現 /
  記事に無い日付・競合名 を中心に点検。

## 出力（記録）

- 修正・削除を行った item にだけ `grounding` を付ける（clean な item には付けない）:
  ```json
  "grounding": { "verified": false, "flags": 2, "notes": ["MMLU 値は記事に無く削除", "『最速』を『高速』に緩和"] }
  ```
  - `flags` = その item で修正/削除した主張の数。`notes` は簡潔に（各 1 行、任意）。
- 全 item の `flags` 合計を、その日の `stats.grounding_flags`（integer）に記録する。
- フロントは `grounding.verified === false` の item に ⚠️ マークを表示する（読者が「要確認」と分かる）。

## 心構え

事実誤りを 1 件出すより、数値を 1 つ落として「やや薄い要約」になる方が良い。
**正確さ > 情報量**。特に公開・自動投稿される文章では妥協しない。
