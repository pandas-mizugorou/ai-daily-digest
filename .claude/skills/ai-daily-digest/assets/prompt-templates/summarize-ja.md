# 日本語要約 + 図解情報生成プロンプト

選定された Top 5-7 件にのみ適用。コスト削減のため**選定後**にのみ呼ぶ。

要約と**同じ呼び出し**で `figure` も生成する（呼び出し回数を増やさない）。

## 設計目的

ユーザーは原則として**元記事を開かない**。`summary_ja` + `key_points_ja` + `figure` の 3 点セットで、記事の内容を**正確かつ完全に把握できる**ことを目標とする。情報量を増やしつつ、構造化で読みやすさを保つ。

## プロンプト本文

```
以下の英語記事情報から、ダイジェストサイト掲載用の日本語要約と「図だけで記事の内容が完結理解できる小図解」情報を作成してください。

【設計目標】
読者は原則として元記事を開かない。summary_ja + key_points_ja + figure の 3 点だけで、記事の内容を**正確かつ完全に**理解できる必要がある。**情報量を確保**しつつ**構造化で読みやすさを担保**する。

【入力】
- title: <記事タイトル>
- source: <ソース名>
- summary_en: <英語要約>
- raw_excerpt: <記事本文の冒頭 2500 字>
- url: <URL>

【出力 JSON】
{
  "title_ja": "<記事タイトルの日本語訳。固有名詞・製品名はカタカナ化せず英語のまま (例: Claude, GPT-4, Llama 3)>",
  "summary_ja": "<350-500 字、4-6 文の日本語要約。元記事を読まずとも内容が完結する密度>",
  "key_points_ja": [
    "<50-120 字、具体的な数字 + 固有名詞 + 結論を含む完結文 1>",
    "<50-120 字、具体的な数字 + 固有名詞 + 結論を含む完結文 2>",
    "<50-120 字、具体的な数字 + 固有名詞 + 結論を含む完結文 3 (最大 6 個)>"
  ],
  "tags": ["<記事のタグ 2-4 個 (英小文字、ハイフン区切り)>"],
  "figure": <summary-card オブジェクト or 省略>
}

【summary_ja のルール — 重要】
- **350-500 字 / 4-6 文**を必ず守る (短すぎ厳禁)
- 構成 (この順番が望ましい):
  (1) リード文: 主役 + 何が起きたか + 主要数字 (1 文目)
  (2) 詳細: 具体的な数字 / 条件 / 対応範囲 / ベンチマーク結果 (1-2 文)
  (3) 背景: なぜ今このニュースか / 業界文脈 / 競合との関係 / 直近の流れ (1 文)
  (4) 影響: 読者にとっての意味 / 何ができるようになるか / 活用方法 (1 文)
- 「である調」で書く。「〜と発表した」「〜が公開された」
- 推測・主観・"〜と期待される" は避ける
- 元記事に書いていない情報は補わない
- 出力は JSON のみ。前置きや末尾の説明は不要

【key_points_ja のルール】
- 各 50-120 字、3-6 個 (3 個以上必須)
- 必ず**具体的な数字または固有名詞**を含める
- 単なる事実列挙ではなく「事実 + 数字 + 結論 / 影響」まで言及
- 例: ❌「性能が向上」 ⭕「MMLU が 82.4 から 94.4 へ 12 ポイント向上、推論系 SoTA を更新し、ツール使用時の精度改善が顕著」
- 各項目は完結文 (途中で切れない)

【figure (summary-card) のルール — 詳細は figure-design.md を参照】

目標: **図だけで記事の内容が完結理解できる**こと。元記事を読まずとも図解だけで「何が・どう・なぜ・どう活かせるか」がすべて伝わる。

省略条件: 記事に具体的な数字・固有名詞・事実が乏しく、points を 3 個埋める具体性がない場合は figure を省略。

形式 (1 種のみ — summary-card、5 セクション構造):
{
  "type": "summary-card",
  "caption": "<図解の主題、≤30字、任意>",
  "alt": "<60-280字、必須。全セクションを文章として読める形で>",
  "data": {
    "headline": "<20-80字、必須。主役+核心事実+主要数字を 1 行に圧縮>",
    "tldr": "<60-180字、必須。3 行で全体把握できる短文。headline の繰り返しではなく補完情報>",
    "points": [
      {
        "icon": "<絵文字1-2字、任意>",
        "label": "<2-12字、観点ラベル>",
        "value": "<1-36字、中心となる値・結論>",
        "note": "<≤50字、差分・条件・単位 — 極力書く>",
        "description": "<40-160字、詳細・背景・具体例 — 極力書く>",
        "tone": "default | primary | success | warning | danger | info"
      },
      ... (3-6 件、必須)
    ],
    "context": "<60-200字、強く推奨。業界文脈・なぜ今・競合との関係>",
    "impact": "<30-160字、強く推奨。読者にとっての意味・どう活かせるか>"
  }
}

figure 必須事項:
- headline は抽象的にせず具体的に (主役名 + 主要数字を含める)
- tldr は headline で書ききれない主要事実を**補完**する (繰り返しは禁止)
- points は 3-6 個、各 description (40-160字) を**極力埋める**
  - description には「なぜそうなったか / 何ができるようになるか / 具体例」まで踏み込む
- context は単なる前置きではなく洞察を: 「直近 6 ヶ月で〜」「競合の〜と異なり」のように業界文脈を含める
- impact は「で、結局どう活かせるの？」に答える
- 記事に書いていない数字・事実を創作しない

tone の使い分け:
- success: 性能向上、新機能、GA、+◯%、改善、対応開始
- warning: 遅延、限定 preview、警告、制限あり
- danger: 撤退、廃止、Deprecated、価格上昇
- info: 情報、変更点、客観的事実、価格据え置き
- primary: 強調したい主役 (その記事で最も注目すべき項目)
- default: 中立
```

## モデル選定

要約は密度 + 品質が必要なので Claude Sonnet を推奨（routine 環境では自動的に親モデルを使用）。

## エラー時のフォールバック

- summary_en だけで raw_excerpt が取れていない場合: summary_en + WebSearch 補強で 350 字以上を確保
- 元記事が日本語ソース（Qiita / Zenn）の場合: title_ja = title、summary_ja は元記事から再構成 (350 字以上)
- summary_ja が 250 字未満になった場合: 当該記事を選定から除外（密度不足）
- figure 生成だけ失敗した場合: figure を省略して push（要約が成立していれば記事は載せる）
- description が複数記事で空になる場合: そのバッチを再生成 (description は基本必須扱い)
