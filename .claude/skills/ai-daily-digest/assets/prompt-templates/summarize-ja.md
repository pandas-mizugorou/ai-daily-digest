# 日本語要約 + 図解情報生成プロンプト

選定された Top 5-7 件にのみ適用。コスト削減のため**選定後**にのみ呼ぶ。

要約と**同じ呼び出し**で `figure` も生成する（呼び出し回数を増やさない）。

## 設計目的

ユーザーは原則として**元記事を開かない**。`summary_ja` + `key_points_ja` + `figure` の 3 点セットで、記事の内容を**正確かつ完全に把握できる**ことを目標とする。

特に `figure` は**視覚で直感的に分かる図解**を 4 型から記事の性質に応じて選ぶ。情報量を増やしつつ、視覚化で読みやすさを担保する。

## プロンプト本文

```
以下の英語記事情報から、ダイジェストサイト掲載用の日本語要約と「視覚で直感的に内容が分かる図解」情報を作成してください。

【設計目標】
読者は原則として元記事を開かない。summary_ja + key_points_ja + figure の 3 点だけで、記事の内容を**正確かつ完全に**理解できる必要がある。**情報量を確保**しつつ**視覚化で直感性を担保**する。

【入力】
- title: <記事タイトル>
- source: <ソース名>
- summary_en: <英語要約>
- raw_excerpt: <記事本文の冒頭 2500 字>
- url: <URL>

【出力 JSON】
{
  "title_ja": "<記事タイトルを日本語で『一目で内容が分かる意訳』にしたもの。直訳厳禁。40-70 字目安。主役 (誰が) + アクション (何をした) + 主要数字または固有名 を含める。固有名詞・製品名はカタカナ化せず英語のまま (例: Claude, GPT-4, Llama 3, SpaceX, Hugging Face)>",
  "summary_ja": "<350-500 字、4-6 文の日本語要約>",
  "key_points_ja": [
    "<50-120 字 × 3-6 個>"
  ],
  "tags": ["<記事のタグ 2-4 個 (英小文字、ハイフン区切り)>"],
  "figure": <4 型のいずれかオブジェクト、必須>
}

【title_ja のルール — 重要】
- **直訳禁止**。原題が "Higher usage limits for Claude and a compute deal with SpaceX" のような場合、
  「Claude のより高い使用制限と SpaceX との計算資源契約」のような直訳ではなく、
  「Anthropic、SpaceX と 22 万 GPU 契約 — Claude Code レート制限を 2 倍に引き上げ」のように、
  本文を 1 行読めば内容が把握できる**意訳の見出し**にする
- **目安 40-70 字** (短すぎ/長すぎ厳禁。ダイジェストカード上段に収まる長さ)
- 構成 (この順番が望ましい):
  (1) 主役 (誰が): 例 "Anthropic"、"OpenAI"、"Google DeepMind"
  (2) アクション (何をした): 例 "発表"、"公開"、"GA"、"買収"、"提携"
  (3) 核となる数字 / 固有名: 例 "Opus 5"、"22 万 GPU"、"+12pt"、"SoTA 更新"
  (4) (任意) 補足の含意: 例 "—Claude Code レート制限 2 倍へ"
- 固有名詞・製品名・モデル名は英語表記を保持: Claude / GPT-4 / Llama 3 / Gemini / Sonnet / Opus / SpaceX / Hugging Face / Anthropic / OpenAI 等
- 数字は半角で、単位は記事の表記を尊重 (`200K` `1M` `94.4` `+12pt` `5x`)
- 主観・推測語禁止 ("革命的", "驚異的", "期待される" 等)
- 副題区切りは「——」(全角ダッシュ) または「：」を使用
- 元記事に書かれていない情報・数字は追加しない
- フロントは `lang === "en"` のときカード上段に **title_ja のみ**を表示する。読者が見る唯一のタイトルなので、原題依存の表現を避け単独で完結させる

【summary_ja のルール — 重要】
- **350-500 字 / 4-6 文**を必ず守る (短すぎ厳禁)
- 構成 (この順番が望ましい):
  (1) リード文: 主役 + 何が起きたか + 主要数字 (1 文目)
  (2) 詳細: 具体的な数字 / 条件 / 対応範囲 / ベンチマーク結果 (1-2 文)
  (3) 背景: なぜ今このニュースか / 業界文脈 / 競合との関係 / 直近の流れ (1 文)
  (4) 影響: 読者にとっての意味 / 何ができるようになるか / 活用方法 (1 文)
- 「である調」で書く。「〜と発表した」「〜が公開された」
- 推測・主観・"〜と期待される" は避ける
- 元記事に書いていない情報は補わない
- 出力は JSON のみ。前置きや末尾の説明は不要

【key_points_ja のルール】
- 各 50-120 字、3-6 個 (3 個以上必須)
- 必ず**具体的な数字または固有名詞**を含める
- 単なる事実列挙ではなく「事実 + 数字 + 結論 / 影響」まで言及
- 例: ❌「性能が向上」 ⭕「MMLU が 82.4 から 94.4 へ 12 ポイント向上、推論系 SoTA を更新し、ツール使用時の精度改善が顕著」
- 各項目は完結文 (途中で切れない)

【figure のルール — 詳細は figure-design.md を参照】

目標: **視覚で直感的に**記事の核心が分かること。図を見ただけで「何が・どう変わった・どう活かせるか」が伝わる。

**figure は全記事で必須**。例外なく生成する。数字が乏しい記事や分析・主張系の記事は、4 型のうち最も適した型 (通常は summary-card) で、**記事内に書かれている事実・固有名詞・主張・論点のみ**を使って構造化する。**創作・推測・元記事にない数字の追加は厳禁**。

# 4 つの図解型から記事の性質に応じて 1 つ選ぶ

| 型 | 適用記事 | 視覚の核 |
|---|---|---|
| comparison | 旧 vs 新 / A 社 vs B 社 など対比構造 | 左右並列の横棒グラフ |
| metric-bars | ベンチマーク等 3-7 個の数値指標 | 横棒グラフ一覧 |
| timeline | ロードマップ・段階展開 | 縦タイムライン |
| summary-card | 上記 3 型に当てはまらない場合 | 5 セクション構造化 |

# 型選択フロー

  「旧 vs 新」「A 社 vs B 社」のような対比構造はあるか？
    YES → comparison
    NO  → 数値指標が 3-7 個並んでいるか？
            YES → metric-bars
            NO  → 時系列イベントが 3-7 個あるか？
                    YES → timeline
                    NO  → summary-card

判断基準: **同じ記事を summary-card でも書ける場合、視覚化が強い 3 型 (comparison / metric-bars / timeline) を優先する**。直感性を優先するため。

**数字が乏しい記事 (分析・主張・解説系) の取り扱い**:
- 必ず summary-card を使い、figure を生成する
- `points[]` には「主張」「論点」「根拠」「対象」「結論」など、記事に書かれている事実・固有名詞を 3-6 個に整理する
- `value` には数字以外でも記事内の固有名詞・キーフレーズを置く (例: 「EvalEval 提案」「主要ベンチマーク 4 種」)
- `tldr` / `context` / `impact` も、記事に書かれた範囲で必ず埋める

# 各型の必須スキーマ

## comparison (対比型)
{
  "type": "comparison",
  "caption": "<≤30字、任意>",
  "alt": "<60-280字、必須>",
  "data": {
    "headline": "<16-80字>",
    "before": { "label": "<比較元名>", "sublabel": "<副題、任意>" },
    "after":  { "label": "<比較先名>", "sublabel": "<副題、任意>" },
    "metrics": [
      {
        "icon": "<絵文字、任意>",
        "label": "<指標名 2-16字>",
        "before": "<旧値>",
        "after": "<新値>",
        "before_pct": <0-100、バー長>,
        "after_pct": <0-100、バー長>,
        "delta": "<差分 例: +12pt / 5x / 据え置き>",
        "delta_tone": "success | warning | danger | info | default",
        "note": "<条件・補足、任意>"
      },
      ... 3-6 件
    ],
    "narrative": "<60-200字、強く推奨>",
    "impact": "<30-160字、強く推奨>"
  }
}

## metric-bars (横棒グラフ型)
{
  "type": "metric-bars",
  "caption": "<≤30字、任意>",
  "alt": "<60-280字、必須>",
  "data": {
    "headline": "<16-80字>",
    "scale": { "max_label": "<≤24字、任意>", "unit": "<≤16字、任意>" },
    "bars": [
      {
        "icon": "<絵文字、任意>",
        "label": "<指標名 2-24字>",
        "value": "<実測値 例: 94.4 / 95.2 pt>",
        "pct": <0-100、必須>,
        "baseline_pct": <0-100、比較基準 任意>,
        "baseline_label": "<比較基準名 任意>",
        "delta": "<差分 任意>",
        "tone": "default | primary | success | warning | danger | info",
        "note": "<1 行補足 ≤100字、任意>"
      },
      ... 3-7 件
    ],
    "narrative": "<60-200字、強く推奨>",
    "impact": "<30-160字、強く推奨>"
  }
}

## timeline (タイムライン型)
{
  "type": "timeline",
  "caption": "<≤30字、任意>",
  "alt": "<60-280字、必須>",
  "data": {
    "headline": "<16-80字>",
    "events": [
      {
        "when": "<2-24字 例: 2026/05 / Q3 2026 / 本日>",
        "label": "<2-50字、イベント名>",
        "description": "<40-140字、強く推奨>",
        "status": "past | now | upcoming",
        "tone": "default | primary | success | warning | danger | info"
      },
      ... 3-7 件、時系列順
    ],
    "narrative": "<60-200字、強く推奨>",
    "impact": "<30-160字、強く推奨>"
  }
}

## summary-card (フォールバック型 — 5 セクション構造)
{
  "type": "summary-card",
  "caption": "<≤30字、任意>",
  "alt": "<60-280字、必須>",
  "data": {
    "headline": "<20-80字>",
    "tldr": "<60-180字>",
    "points": [
      {
        "icon": "<絵文字、任意>",
        "label": "<2-12字>",
        "value": "<1-36字>",
        "note": "<≤50字>",
        "description": "<40-160字、極力埋める>",
        "tone": "default | primary | success | warning | danger | info"
      },
      ... 3-6 件
    ],
    "context": "<60-200字、強く推奨>",
    "impact": "<30-160字、強く推奨>"
  }
}

# 共通の必須事項

- alt は 60-280 字必須 (視覚障害ユーザー用、図に表れている全情報を文章化)
- headline は具体的に (主役名 + 主要数字を含める、抽象表現禁止)
- narrative / impact は強く推奨 (単なる前置き禁止、業界文脈と読者影響を踏み込む)
- 数値はバーの長さに正規化:
  * comparison: 同じ指標内で max=100 として before_pct / after_pct を算出
  * metric-bars: 各 bar の値が % なら直接使用、単位混在なら最大値=100 で正規化
  * baseline_pct は前世代・他社のスコア (任意)
- 色 (tone / delta_tone) で意味を区別: 改善 = success / 悪化 = danger / 据え置き = info / 強調 = primary
- 記事に書いていない数字・事実は禁物 (創作禁止)

# 創作禁止 (省略条件は廃止)

figure は全記事で必須。ただし以下は**絶対禁止**:
- 記事に書かれていない数字・パーセント・スコアの追加
- 元記事にない比較対象・年代・地名の追加
- 推測に基づく将来予測・期待値・見込み

このとき:
- 数字が乏しい場合 → summary-card で記事内の事実 (固有名詞・主張・論点) を構造化
- 比較対象が記事にない場合 → comparison ではなく metric-bars / summary-card を選ぶ
- ロードマップ情報がない場合 → timeline は使わない
- どうしても 3 件以上の有効ポイントが埋まらない場合のみ → 当該記事を選定から除外 (要約品質不足として扱い、figure は省略しない)

# tone の使い分け

- success: 性能向上、新機能、GA、+◯%、改善、対応開始
- warning: 遅延、限定 preview、警告、制限あり
- danger: 撤退、廃止、Deprecated、価格上昇
- info: 情報、変更点、客観的事実、価格据え置き
- primary: 強調したい主役 (その記事で最も注目すべき項目)
- default: 中立
```

## モデル選定

要約は密度 + 品質が必要なので Claude Sonnet を推奨（routine 環境では自動的に親モデルを使用）。

## エラー時のフォールバック

- summary_en だけで raw_excerpt が取れていない場合: summary_en + WebSearch 補強で 350 字以上を確保
- 元記事が日本語ソース（Qiita / Zenn）の場合: title_ja = title、summary_ja は元記事から再構成 (350 字以上)
- summary_ja が 250 字未満になった場合: 当該記事を選定から除外（密度不足）
- figure 生成だけ失敗した場合: **機械フォールバック figure を必ず生成する**。既存の `title_ja` / `summary_ja` / `key_points_ja` から下記ルールで機械的に summary-card を組み立てる。**選定除外は絶対にしない**（重要記事を figure 失敗で落とすのは本末転倒）
- 4 型のどれも適合しないが事実が豊富な場合: summary-card にフォールバック

### 機械フォールバック figure の組み立て規則

LLM が記事から構造化情報を抽出できなかった場合に限り、以下のテンプレで summary-card を必ず埋める。新しい事実は一切追加せず、既存フィールドの転記のみで完結させる:

```jsonc
{
  "type": "summary-card",
  "alt": "<title_ja を含めて 60 字以上の説明文。例: 『<title_ja> の概要を <source_label> から伝える要約カード。記事の主要ポイント N 個を整理した。』>",
  "data": {
    "headline": "<title_ja をそのまま (20-80 字に収まらない場合は末尾を省略記号で詰める)>",
    "tldr": "<summary_ja の冒頭 2 文。60-180 字を超える場合は文単位でトリム>",
    "points": [
      // key_points_ja の各要素を 1 件ずつ変換、3-6 個
      {
        "label": "ポイント1",
        "value": "<key_points_ja[i] の先頭 36 字以内 (記号除去)>",
        "description": "<key_points_ja[i] の全文 160 字以内>",
        "tone": "default"
      }
      // ... 同様に「ポイント2」「ポイント3」...
    ],
    "context": "<source_label + ' / ' + published_at>",
    "impact": "<key_points_ja の末尾要素を 30-160 字でトリム、または省略>"
  }
}
```

このフォールバックは「視覚化は弱いが情報は保つ」設計。**通常時は専用 figure (comparison / metric-bars / timeline) を優先**し、フォールバックは最後の保険として位置付ける。
