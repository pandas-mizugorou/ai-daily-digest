# 図解 (`figure`) 設計プロンプト — 図だけで元記事を読まずに内容を完結理解できる

## 目標

ユーザーは原則として**元記事を開かない**。生成されたカードと図解だけで、AI / 生成 AI ニュースの**核心・詳細・背景・影響**を正確に理解したい。

そのために図解 `summary-card` は**5 セクション構造**を持つ「視覚化された解説記事」として設計する。**情報量を増やしつつ、セクション分けで読みやすさを担保**するのが核心方針。

## 5 セクション構造

```
┌─ HEADLINE ──────────────────────────────┐  20-80字、主役+核心事実
│ Claude Opus 5: 推論 1.4倍 + 1M tokens   │
├─ TL;DR ─────────────────────────────────┤  60-180字、3行で全体把握
│ Anthropic が新フラッグシップ Opus 5 を…  │
├─ DETAILS (points) ──────────────────────┤  3-6項目、各項目に詳細説明
│ 📈 性能  / MMLU 94.4 / 前世代 +12pt     │
│   推論系全般で SoTA。HumanEval も 95.2  │
│   で過去最高、特にツール使用時の…         │
│ 📜 コンテキスト / 1M tokens / 200K→1M  │
│   …                                     │
├─ CONTEXT ───────────────────────────────┤  60-200字、業界文脈・位置付け
│ 直近 6 ヶ月のフロンティアラボ競争で…       │
├─ IMPACT ────────────────────────────────┤  30-160字、読者への意味
│ コードベース全体を 1 リクエストで…         │
└─────────────────────────────────────────┘
```

### 各セクションの役割と書き方

| セクション | 役割 | 文字数 | 書き方の核 |
|---|---|---|---|
| `headline` | 1 行で核心を伝える | 20-80字 | 主役 (ベンダー/プロダクト) + 何が起きたか + 主要数字 |
| `tldr` | 3 行で全体を理解させる | 60-180字 / 2-3 文 | headline で書ききれない主要事実を補完。**読者を 10 秒で記事の輪郭に到達させる** |
| `points` (3-6 個) | 詳細な事実を構造化 | 各 100-300字 | label / value / note / **description** の 4 段。description で「なぜそうなったか / 何ができるか」まで踏み込む |
| `context` | 業界文脈・位置付け | 60-200字 | なぜ今このニュースが重要か。直近の業界動向・競合との関係 |
| `impact` | 読者への意味 | 30-160字 | 「で、結局どう活かせるの？」に答える 1-2 文 |

### 階層的に読める設計

- **5 秒で**: `headline` だけ → 何が起きたか把握
- **15 秒で**: + `tldr` → 大枠を理解
- **1 分で**: + `points` → 数字・条件・詳細をすべて把握
- **3 分で**: + `context` + `impact` → 背景・影響まで含めて完全理解

このカードを最後まで読めば**元記事と同等以上の理解**が得られることを目標とする。

## points の `description` 必須化

旧来は `value` + `note` の 2 段だったが、これでは情報密度が不足する。新形式では **`description` (40-160字) で各ポイントを文章で補強**する。

例:
```json
{
  "icon": "📈",
  "label": "性能",
  "value": "MMLU 94.4",
  "note": "前世代 +12pt (82.4 → 94.4)",
  "description": "推論系ベンチマーク全般で SoTA を更新。HumanEval も 95.2 で過去最高、特にエージェントとしてツールを使う設定での精度改善が顕著。",
  "tone": "primary"
}
```

`value` で**結論**、`note` で**差分・条件**、`description` で**詳細・背景・具体例**まで言及するのが理想形。

## 10 原則 — LLM が `figure` を作るとき必ず守る

1. **元記事を読まなくても完結すること**: 5 セクションに「事実 + 数字 + 背景 + 影響」を漏れなく配置
2. **`headline` は核心を 1 行に**: 主役 + 主要数字を必ず含める
3. **`tldr` は記事を 3 行に圧縮**: headline の繰り返しではなく、補完情報を入れる
4. **`points` は 4 段構造**: `label` / `value` / `note` / `description` を全て埋めるのが基本。description は省略可だが**極力埋める**
5. **数字 + 固有名詞 + 結論を全段で**: 抽象表現を避け、具体的事実を書く
6. **`context` は単なる前置きではなく洞察**: 業界文脈・競合との関係・なぜ今かを書く
7. **`impact` は当たり前を書かない**: 「性能が向上した」ではなく「コードベース全体を 1 リクエストで読める設計が現実的に」
8. **記事に書いていない情報は禁物**: 創作禁止。事実に忠実に
9. **色 (`tone`) で意味を区別**: 増加/GA = `success`、警告 = `warning`、撤退/廃止 = `danger`、情報 = `info`、強調 = `primary`
10. **記事に具体性が乏しいときは figure を省略**: 数字も固有名詞もないニュースに無理やり作ると事実誤りリスク

## スキーマ

```json
{
  "type": "summary-card",
  "caption": "<図解の主題、≤30字、任意>",
  "alt": "<60-280字、必須。全セクションを文章として読める形で>",
  "data": {
    "headline": "<20-80字、必須>",
    "tldr": "<60-180字、必須>",
    "points": [
      {
        "icon": "<絵文字、任意>",
        "label": "<2-12字>",
        "value": "<1-36字>",
        "note": "<≤50字、極力書く>",
        "description": "<40-160字、極力書く。各ポイントの詳細・背景・具体例>",
        "tone": "default | primary | success | warning | danger | info"
      },
      ... (3-6 件、必須)
    ],
    "context": "<60-200字、強く推奨>",
    "impact": "<30-160字、強く推奨>"
  }
}
```

### アイコン (絵文字) の選び方

| 観点 | 絵文字 |
|---|---|
| 性能・スコア | 📈 🎯 ⭐ 🏆 |
| 速度・レイテンシ | ⚡ ⏱️ 🚀 |
| コスト・価格 | 💰 💵 💸 |
| コンテキスト・容量 | 📜 📦 💾 |
| ツール・SDK | 🔧 🛠️ |
| API・接続 | 🌐 🔌 |
| 公開・GA | ✅ 🎉 |
| 警告・制限 | ⚠️ ⏸️ |
| 廃止・撤退 | 🚫 ❌ |
| データ・統計 | 📊 📉 |
| 環境・対応 | 🖥️ 💻 |
| 言語 | 🐍 (Python) 📘 (TS) |
| ベンチマーク | 🏆 |
| ライセンス | 📄 |
| セキュリティ | 🔒 |
| 時間・スケジュール | 📅 ⏰ |
| ユーザー・チーム | 👥 |
| ドキュメント | 📖 |

迷ったら `📊` でよい。

### tone の使い分け

| tone | 適用例 |
|---|---|
| `success` | 性能向上、新機能追加、GA、+◯%、改善、対応開始 |
| `warning` | 遅延、限定 preview、警告、制限あり |
| `danger` | 撤退、廃止、Deprecated、脆弱性、価格上昇 |
| `info` | 情報、変更点、客観的事実、価格据え置き |
| `primary` | 強調したい主役（その記事の中で最も注目すべき項目） |
| `default` | 中立、上記いずれでもない |

## Bad 例（避けるべき）

### Bad 1: `description` 省略で情報量不足
```json
{ "label": "性能", "value": "94.4", "note": "+12pt" }  // ❌ なぜ重要か / 何ができるか不明
```
→ Good: `description: "推論系ベンチマーク全般で SoTA を更新。HumanEval も 95.2 で過去最高..."`

### Bad 2: `tldr` が `headline` の繰り返し
```json
{
  "headline": "Claude Opus 5 発表",
  "tldr": "Anthropic が Claude Opus 5 を発表した。"  // ❌ 情報追加なし
}
```
→ Good: tldr は headline で書ききれない主要事実を補完する

### Bad 3: `context` が空 / 一般論
```json
{ "context": "AI モデルの進化が続いている" }  // ❌ 当たり前
```
→ Good: `"直近 6 ヶ月で OpenAI / Google も長文対応モデルを発表したが、長文時のレイテンシが課題だった..."`

### Bad 4: `impact` が記事内容の繰り返し
```json
{ "impact": "性能が向上する" }  // ❌ 価値が伝わらない
```
→ Good: `"コードベース全体を 1 リクエストで読み込むエージェント設計が現実的に..."`

### Bad 5: 創作した数字
記事に書いていない数値・固有名詞を `description` や `context` に入れない。記事の事実を超えた推測は厳禁。

## 省略の基準

以下のいずれかに当てはまる場合は `figure` を**含めない**:

- 記事に具体的な数字や固有名詞が一切ない
- `headline` を作っても抽象的にしかならない
- `points` を 3 個埋められるだけの具体性が記事にない
- 質的な解説記事で量的な事実に落とし込めない

無理に作ると **「事実誤り」リスク**が上がる。ない場合は省略が正解。
